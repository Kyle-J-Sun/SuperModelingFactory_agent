# Known gotchas ‚Äî a history log, not a current-state list

Every entry is tagged with when it was true. **Before repeating one of these to the user, check whether the installed version is at or past the fix** (introspect the relevant source ‚Äî `inspect.getsource`) and say so either way: "this was real before v0.3.X, confirmed fixed in your current source" is a better answer than silently re-warning about something already closed, and better than silently assuming it's fixed without checking.

## Reject inference direction bugs (fixed 0.3.3)

Root cause across all of these: `Sample.Reject_Infer`'s inferrers were written assuming "higher score = lower risk," while `RejectInferencePipeline` fed them `prescore_prob` = P(bad) ‚Äî the opposite polarity ‚Äî with no parameter to reconcile the two.

- **HardCutoffInferrer** ‚Äî `target = (score < cutoff)` under P(bad) input flags the *safest* rejects as bad and vice versa. Symptom that gave it away: inferred reject bad-rate came out *lower* than the real approved bad-rate, which makes no business sense (rejects should look worse, not better).
- **FuzzyAugmentInferrer** ‚Äî rejected pseudo-label was `1 - score` (wrong direction, and continuous rather than a proper good/bad split); approved-row reweighting formula additionally suppressed toward zero any real row the prescore had misjudged, amplifying the prescore's own bias instead of correcting for it.
- **ParcelingInferrer** ‚Äî approved population binned via `qcut` (equal-frequency), rejected population binned via `cut` (equal-width, over the *rejected* population's own score range) ‚Äî two unrelated bin systems mapped onto each other by index, effectively assigning near-random bad rates.
- **Pipeline-level (`_train_and_evaluate_models`)** ‚Äî `split_col` mode set `val = approved.copy()`, i.e. validation was literally the training population; every non-weighted method's `train_AUC` and `validation_AUC` came out identical to 6 decimal places, and GBM early stopping had no real holdout to stop against.
- **Continuous pseudo-labels got hard-binarized at `>0.5`** downstream in the pipeline, compounding whichever of the above was already wrong.
- **SimpleAugmentInferrer** hardcoded `np.random.seed(42)`, polluting the global RNG state and ignoring `cfg.random_state` (lower severity ‚Äî the method is "assign the approved bad-rate at random" by design, so the direction bugs above don't apply to it, but reproducibility did).

**Fix:** `score_direction` parameter added to every inferrer + `ri_score_direction` (`"high_bad"`/`"high_good"`) on the pipeline config; `_default_hard_cutoff`'s percentile flips with direction; fuzzy rewritten to the textbook form (approved weight 1, rejected duplicated into weighted good/bad rows); parceling given shared `parcel_edges_`; `ri_validation_frac` added for a genuine holdout; SimpleAugment moved to an instance RNG.

**Durable lesson:** any score-consuming module needs its polarity convention pinned and checked against what's actually fed in. These bugs threw no exceptions ‚Äî they just produced confidently-wrong, business-plausible-looking numbers (a "hard cutoff" method assigning rejects a *lower* bad rate reads as surprising but not obviously broken until you check the arithmetic).

## Weighted screening bugs (introduced 0.3.7, fixed 0.3.14)

- **Correlation silently blind to NaNs.** `_weighted_pearson_corr_matrix` ran `np.cov(..., aweights=w)` directly on raw (possibly-NaN) columns; any NaN anywhere in a column propagates `cov` to NaN, which then got `np.nan_to_num(nan=0.0)`'d ‚Äî i.e. any feature pair where either column has missing values reads as *uncorrelated*, and both survive correlation dedup untouched. On this project's real feature matrices (20‚Äì35% NaN from cleaned sentinel values) this made the correlation stage close to a no-op for most pairs. Confirmed empirically with a synthetic 30%-NaN pair at true correlation ‚âà0.999: both columns survived pre-fix. The weighted path also ignored `corr_use_woe_bins` (the unweighted path respected it).
- **Silent "everything survives" fallback.** Every stage (PSI/IV/corr) used `current = keep or current` ‚Äî if a threshold eliminated every remaining variable, the function quietly returned the untouched input set with no warning and no trace in the summary table.

**Fix (0.3.14):** `corr_nan_policy` (`"pairwise"` default ‚Äî complete-case overlap per pair; `"median_fill"`; `"raise"`); weighted corr now also respects `corr_use_woe_bins`; `on_empty_stage` (`"keep_all_warn"` default ‚Äî same fallback behavior, but now `warnings.warn`s and appends a `*_fallback` row to `.summary`; `"raise"` for strict use).

**Durable lesson:** a covariance/correlation computation over a real feature matrix needs an explicit NaN policy stated out loud ‚Äî the numpy default ("propagate to NaN, then someone downstream coerces NaN to 0") is silently equivalent to "assume uncorrelated," which is the least safe possible default for a dedup step. Separately: any "keep everything" fallback is only a safety net if it's loud: a silent one is indistinguishable from "the filter worked."

## Mixed categorical + numeric IV screening KeyError (0.3.14, fixed 0.3.15)

- **Trigger:** `selection_enabled=True` + shared WOE bins (`iv_use_woe_bins=True`) + `categorical_features` passed to `MonotoneWOEBinner(cate_feats=...)` + at least one categorical feature survives IV cut alongside numeric features in the same `high_iv_varlist`.
- **Symptom:** `KeyError: "['MIN', 'MEAN', 'MAX'] not in index"` in `WOE_Engine_Feature_Patch._screening_summary_from_engine` (or the parallel path in `Feature_Insights.get_var_analysis_report`) when merging `proc_means_by_grp` output. Pure-numeric feature sets unaffected.
- **Root cause:** `proc_means_by_grp` melts numeric + object columns together; pandas `describe()` on the mixed melt returns `count/unique/top/freq` for object columns, not `MIN/MEAN/MAX`.
- **Fix (0.3.15):** `Distribution_Tool.proc_means_for_screening` splits by dtype before calling `proc_means_by_grp`; categorical rows get `min/mean/max` as NaN.

**Durable lesson:** any batch stats helper that assumes numeric `describe()` columns will break the moment categorical features share a melt with numerics ‚Äî split by dtype or reindex defensively.

## RejectInferencePipeline weighted GBM training was a no-op (fixed 0.3.16)

- **Trigger:** any RI method that emits `_weight` (`fuzzy_augment`, `parceling`) with `train_ri_models=True`. Approved-only methods (`simple_augment`, `hard_cutoff`) are unaffected.
- **Symptom:** none ‚Äî training succeeds, no warning, no error. Model output looks plausible. But the weighted rows have zero effect: swapping in any other weight vector (all-ones, random, business rebalance) trains an identical model to the true `_weight`.
- **Root cause (`Pipeline/reject_inference.py` L623 in 0.3.15):** `fit_kwargs = {"wgt": weight}` passed to `GradientBoostingModel.fit`. The fit signature is `(x, y, valx, valy, init_score=None, sample_weight=None, eval_sample_weight=None, sample_weight_eval_set=None, **kwargs)` ‚Äî `wgt` is not a real parameter and is silently absorbed by `**kwargs` and dropped. `GradientBoostingModel` uses `sample_weight`. Old SMF versions did expose `wgt=` on LGB fits, but the API was unified to `sample_weight` in an earlier version and this pipeline site was never updated.
- **Fix (0.3.16):** `fit_kwargs = {"sample_weight": weight}`. Regression test `SuperModelingFactory_pytest/test_reject_inference_weighted.py` intercepts `GradientBoostingModel.fit` and asserts (a) `sample_weight` is in the observed kwargs, (b) `wgt` never is, (c) the vector length matches the train row count.

**Durable lesson:** any `**kwargs`-accepting fit / call site where a Pipeline hand-rolls the kwarg dict is a silent-drop trap when the underlying signature evolves. When SMF unifies a param name (as with `wgt` ‚Üí `sample_weight`), grep every callsite in `Pipeline/*.py` at the same time, not just the `Model/*.py` definition ‚Äî the compiler will not catch drift into `**kwargs`. `require_params(fit, "sample_weight")` in the pytest suite would catch this class of regression.

## Missing-rate screening stage gap (fixed 0.3.15)

- **Before 0.3.15:** `FeatureScreenConfig` and `FeatureValidationPipelineConfig` had no `missing_rate_threshold`; screening ran PSI ‚Üí IV ‚Üí corr only. `missing_rate` appeared in IV tables but was not a pre-PSI filter stage.
- **Fix (0.3.15):** `missing_rate_threshold: float | None = None` on both configs; when set, stage 0 runs on INS before PSI, drops features above threshold, persists `missing_rate_table` / `missing_rate_dropped` on `WeightedScreenResult` and FVP selection outputs.

## `apply_woe` O(n¬≤) performance (fixed before 0.3.3)

`WOE_Monotone_Binner.apply_woe` inserted new WOE columns one at a time via `df[col] = ...`, each insertion forcing pandas to reallocate its internal block manager. On wide batches (100‚Äì250+ new columns per call) this dominated runtime and was the confirmed (via `py-spy` stack sampling of a live, stuck process) root cause of a 3,380-candidate-feature validation run blowing through a 12-hour cell timeout with 14/14 batches still not finished. Fixed by collecting the new columns and adding them via a single `pd.concat(axis=1)`.

**Durable lesson:** if a WOE/feature-assembly step scales worse than linear with column count, suspect column-by-column DataFrame mutation before suspecting the statistics.

## Persistence

- **Always load SMF-saved models via `Core.load_model(path, return_metadata=True)`.** A bare `pickle.load` on an SMF model artifact raises a confusing `No module named '1'`-shaped error ‚Äî SMF wraps saved models in its own versioned container, not a plain pickle of the estimator.
- **`LRMaster` used to pickle its full training frame (`_data`)** ‚Äî tens to hundreds of MB, and literally row-level customer data riding along inside a model artifact (a real governance issue, not just bloat). Fixed via `__getstate__`/`__setstate__` dropping `_data` on save. Consequence: `get_statsmodel_summary` / `get_aic` / `get_bic` called on a *loaded* model now require an explicit `data=` kwarg and raise a clear `ValueError` telling you so if you omit it ‚Äî this is intentional, not a regression.
- **`CreditModelPipelineConfig` had no `save_models` field at all** before it was added ‚Äî the `models/` output directory was created either way (dead giveaway that something's missing, not that nothing was supposed to be there) but stayed empty until the field landed.

## Evaluation-table labeling

- **`model_perf_compare`'s `sample_name` was hardcoded to `"oot"`** regardless of what population was actually being scored ‚Äî every table it powers (including every one of `ScoreComparisonPipeline`'s global and group-by outputs) had its `index`/scope column mislabeled `"oot"` even for the full sample or an arbitrary subgroup. Fixed by adding an explicit `sample_name` parameter; `ScoreComparisonPipeline` now passes `"global"` at every internal call site, and the bare function's own default changed to the neutral `"all"`. If anything calls `model_perf_compare` directly without passing `sample_name`, its output labeling changed meaning across this version bump ‚Äî check call sites, not just the pipeline.

## Explainability

- **Owen values deliberately skip `xgb`** ‚Äî `_run_explainability`'s Owen branch is gated `name != "xgb"`. By design, not a bug; don't "fix" it without finding out why first.
- **LIME was fully hand-rolled through 0.3.6**, absorbed natively into `Explainability.Model_Explainer` (`lime_explain_instance`, `lime_global_importance`, with built-in missing-value handling) as of **0.3.13**. This is the concrete precedent for the escalation ladder in `SKILL.md`: a hand-built workaround (including its NaN-median-fill fix for the fact that raw/non-WOE features commonly contain NaNs, which vanilla LIME's sampler can't handle ‚Äî `scipy.stats.truncnorm: scale must be positive`) can be quietly absorbed upstream. **Check `ModelExplainer` for a native method before hand-rolling an explainer again on any version ‚â•0.3.13.**
- **`explain_outputs` (the computed SHAP/Owen tables) may not be persisted to disk** even when an `explain/` output directory is created and `explain_models`/`owen_enabled` ran without error ‚Äî the tables can live only in the in-memory `CreditModelPipelineResult` for that kernel's lifetime. If the kernel dies before you persist them yourself (a later cell crashing, an OOM, a LIME NaN blowup), they're gone; recover by reloading the saved model via `Core.load_model(..., return_metadata=True)` (for `feature_cols`/`feature_source`) and recomputing with the same `sample_n`/`background_n`/`random_state`. Check whether the installed version's `_run_explainability` calls a persistence helper (look for something like `persist_explain_outputs` in `Pipeline._common` actually being invoked, not just importable) before assuming this gap is still open.
- **Environment, not SMF:** a long stretch of this project ran with `explain_models=[]` as a standing workaround because the installed `numba` (0.56.4, compiled against numpy<1.24) was ABI-incompatible with the installed `numpy` (2.2.6) ‚Äî `import shap` crashed transitively via numba. `_run_explainability` catches this per-model and returns `{"error": repr(exc)}` rather than failing the whole pipeline, which can look like "explainability isn't configured" rather than "the environment is broken" ‚Äî if a model's `explain_outputs` entry has an `"error"` key, check the environment before the config. Working combination found: `numba==0.66.0` + `llvmlite==0.48.0` + `lime==0.2.0.1` + `scikit-image>=0.25` + `pywavelets>=1.8`, with numpy/shap/lightgbm/xgboost/catboost left untouched.

## Data hygiene that belongs *before* the SMF call, not inside one

Neither of these is an SMF bug ‚Äî both are exactly the kind of requirement that feels like "I'll just hand-filter the DataFrame" until you check whether a Config field already covers it:

- **Negative sentinels in CDC feature/score columns** (`-1`, `-2`, `-100`, observed at ~35% of an entire feature matrix's cells by count) are not valid numeric values. SMF's binners will bin them as real numbers if you feed them in as-is ‚Äî always `NaN` them first, and do it consistently for every column an evaluation touches (score columns included, not just model features).
- **A population that should be excluded from fitting but retained for evaluation** (this project's "FT" cohort ‚Äî a natural-approval segment that bypasses normal model scoring) is handled by `woe_fit_query` (fit-time exclusion) + `extra_eval_datasets` (eval-only inclusion) on the relevant pipeline, once those fields exist on the installed version ‚Äî check before hand-splitting the DataFrame yourself.

## Pipeline hardening batch (fixed 0.3.17)

Four independent HIGH-severity gaps found during a full re-read of the seven Pipeline modules on the 0.3.15 ‚Üí 0.3.16 baseline, all shipped together in 0.3.17. None of them threw a loud exception before the fix; all four either produced silently-wrong outputs or reported an error a long way from the actual root cause.

- **H1 ‚Äî `CreditModelPipeline._fit_models` mutated the user's `cfg.model_params["lr"]`** on the LR training branch. The old line was `lr = LRMaster(params=params or None, standardize=bool(params.pop("standardize", False)) if params else False)`; `params.pop("standardize", False)` ran on the *same* dict already passed to `LRMaster(...)` (via reference), and the truthiness short-circuit made the behavior further order-dependent. Symptom: user's config `{"standardize": True, ...}` had `"standardize"` disappear after `.run()`, breaking a second run or any downstream code that expected the field. Fix: copy first, pop second ‚Äî same pattern the neighboring `_run_lr_search` already used at L901‚Äì902.
- **H2 ‚Äî `oot_col == 0` silently routed every string-flag row to OOT.** Both `CreditModelPipeline._split_data` (L420‚Äì421 pre-fix) and `FeatureValidationPipeline._split_dataset` (L910‚Äì912 pre-fix) used `work[cfg.oot_col] == 0` / `!= 0`. If the column came in as `"0"`/`"1"` strings or `"train"`/`"test"` labels (common Hive/ODPS export artifact), the equality comparison against integer `0` was false for every row ‚Üí `ins_oos` came out empty, `oot` swallowed the whole frame, no exception. Fix: shared `split_oot_by_flag` helper in `Pipeline/_common.py` runs `pd.to_numeric(..., errors="raise")` ‚Äî numeric strings (`"0"`/`"1"`) still work, boolean flags still coerce, but non-numeric content now raises a clear `TypeError` at the split step instead of producing empty training data.
- **H3 ‚Äî `RejectInferencePipeline` implicit prescore path was un-validated.** `_run` triggers prescore training either when `cfg.train_prescore=True` **or** when `cfg.score_col not in data.columns` (implicit path ‚Äî no external score means we have to fit one). `_validate_input` only checked `target_col` presence on the first branch, so the second branch failed deep inside `_fit_prescore` with a positionally confusing `KeyError`. Fix: `_validate_input` now checks `cfg.train_prescore or cfg.score_col not in data.columns` up-front and raises with a message that includes both `train_prescore` and the `score_col` presence status.
- **H4 ‚Äî `RejectInferencePipeline` `oot_frac + ri_validation_frac >= 1` silently leaked OOT into validation.** `_sample_validation_ids` builds `pool = approved[~approved["_smf_ri_row_id"].isin(exclude_ids)]`; when `exclude_ids` (OOT + prior selections) covered everything, `pool` was empty and the code fell back to `pool = approved.copy()` ‚Äî meaning the "held-out" validation set was resampled from the full approved population, including the OOT rows. Metrics on validation and OOT then shared rows, quietly biasing the validation number toward OOT and invalidating any early-stopping decision that used it. Fix: `_validate_ri_approved_config` now enforces `oot_frac + ri_validation_frac < 1` and `oot_frac ‚àà [0, 1)` at `run()` start; the fallback branch itself was left alone because the assertion makes it unreachable under any valid config.

**Durable lessons across all four:**
- Silent equality-based routing on an ambiguously-typed column (`work[col] == 0`) is a landmine any time the column might be a string/boolean; always coerce and validate at the boundary.
- "Fallback to full population" is only safe if it's loud ‚Äî the RI validation fallback and the string-oot-flag routing both silently produced business-plausible-looking but wrong numbers.
- When one code path validates a precondition and a parallel path skips validation, the "unchecked" path is where the confusing error will surface ‚Äî always check every branch that requires the precondition, not just the obvious one.

## Pipeline hygiene batch (fixed 0.3.18)

Three independent MEDIUM-severity gaps found in the same 0.3.15 ‚Üí 0.3.17 review that produced the H1‚ÄìH4 batch, held back for a follow-up release to keep 0.3.17 focused. None crashed loudly on the fix path; each either produced silently-wrong data, silently discarded user intent, or failed at a confusing point far from the real problem.

- **M2 ‚Äî `CreditModelPipeline._fit_woe` copied `warm_start_score_col` with a bare `.to_numpy()` and no length guard.** Both the reused-from-screening branch (L600‚Äì606 pre-fix) and the fresh-fit branch (L697‚Äì704 pre-fix) did `df[cfg.warm_start_score_col] = splits[name][cfg.warm_start_score_col].to_numpy()` to copy the warm-start scores from raw splits into WOE-transformed splits. This relied on WOE transformation preserving the row count exactly; any upstream `.dropna()`, fit-query filter, or reindex on either side would silently corrupt the join ‚Äî one frame's rows attached to another frame's scores, no error, no warning. Fix: new `Pipeline/_common.copy_column_length_checked(dst, src, col, dst_name, src_name)` helper enforces `len(dst) == len(src)` before the positional `.to_numpy()` copy and raises `ValueError` with both frame names if they've drifted. Both `_fit_woe` call sites and the `_reuse_screening_woe` helper now route every warm-start copy through it.
- **M3 ‚Äî `RejectInferencePipeline._run` silently overwrote a user-supplied `cfg.score_col`.** The prescore path fires when `cfg.train_prescore=True` **or** when `cfg.score_col not in data.columns`. In the combined case (`train_prescore=True` *and* `score_col` already present in the input frame) `_fit_prescore` unconditionally wrote its own model's predictions into that column with no warning ‚Äî the user's pre-supplied score column vanished with no trace. Fix: emit a `UserWarning` before calling `_fit_prescore` when the overwrite is about to happen, naming the column and both opt-outs (`train_prescore=False` to reuse, or rename `cfg.score_col` to sidestep the collision). Skipping the prescore fit was rejected because a user who explicitly set `train_prescore=True` *is* asking for a fresh fit ‚Äî the bug is only that the collision was invisible, not that the overwrite was inherently wrong.
- **M4 ‚Äî `_fit_prescore` fed an empty frame to `SampleSplitter.split_df`.** `approved = data[(approved_col==1) & target_col.notna()]` with no guard: if no approved rows had observed targets (mislabeled columns, an all-rejected batch, target-column contract mismatch), `SampleSplitter.split_df` failed deep inside sklearn with a confusing `stratify` error. Fix: `_fit_prescore` now raises a targeted `ValueError` when `len(approved) == 0` with counts of `approved` and `target-observed` rows plus the total row count ‚Äî the message tells the caller whether they have an approved-count problem, a target-null problem, or both. Aligns with the H3 pattern of failing at input validation with diagnostics rather than deep in a downstream call.

**Durable lessons across all three:**
- A positional column copy (`.to_numpy()`, `.values`) is only safe if the length invariant is enforced explicitly ‚Äî pandas won't error on a length mismatch because there is no length mismatch at that call, the mismatch is between the two frames being combined and pandas never sees them together.
- "Overwrite silently on collision" is a defensible default only when the collision itself is impossible; the moment two config paths (`train_prescore`, `score_col`) can both be set, the collision case needs a warning at minimum.
- Any deep-in-sklearn error a Pipeline can predict from its own preflight state should raise early with counts, not later with a stack trace pointing at library internals.

## Pipeline hygiene batch #2 (fixed 0.4.0)

Six independent MEDIUM-severity gaps across five Pipeline modules, shipped together. One of them (**M5**) is a behavior change that flips a default and forced the version bump from a 0.3.19 patch to a 0.4.0 minor. The other five are pure hardening ‚Äî silent-failure closures, one performance rewrite, one preview API.

- **M1 ‚Äî `CreditModelPipeline._run_optuna` fell back to defaults on empty `search_spaces={}`.** The old line was `search_spaces = cfg.optuna_params.get("search_spaces") or self._default_search_spaces()` ‚Äî an explicit empty dict is falsy, so `or` swapped it out for the built-in default. Users passing `optuna_params={"search_spaces": {}}` to say "run no optuna search" got the default search anyway. Fix: `is None` check instead of truthiness. Untangling the same footgun in any newly written pipeline code is now cheap because the pattern is fresh.
- **M5 ‚Äî `ScoreComparisonPipelineConfig.cross_vars` default flipped from `["rating"]` to `[]`.** The old default forced a `rating` column via `_validate_input` and crashed for the (majority) callers who don't have one. In 0.4.0 the default is `[]`, `_validate_input` no longer includes `cross_vars` in the required-columns list, and the run loop soft-skips any `cross_var` that isn't in the input frame with a `_logger.warning`. **Breaking change**: callers who relied on the implicit rating breakdown must now pass `cross_vars=["rating"]` explicitly. This is why the release is 0.4.0 rather than 0.3.19.
- **M6 ‚Äî `ScoreConsistencyUATPipeline` silently coerced object columns to numeric.** `_load_dataframes` ran `pd.to_numeric(df[col], errors="coerce")` on every object column and swapped in the coerced version whenever `notna().any()`. Rating columns like `["A", "B", "1", "2"]` silently lost the alpha buckets to NaN. Fix: new `numeric_coercion_mode` config field with three modes (`"safe"` default = only coerce when ‚â•`numeric_coercion_min_ratio` of non-null values parse, warn otherwise; `"aggressive"` = old 0.3.x behavior but with a warning per lossy column; `"off"` = never coerce). To restore 0.3.x behavior explicitly: `numeric_coercion_mode="aggressive"`.
- **M7 ‚Äî `FeatureValidationPipeline._split_frame` bare-except swallowed stratify failures.** `try: SampleSplitter(...).split_df(...) except Exception: pass` dropped every failure into a naive random split with no diagnostic. Fix: `except Exception as exc` + `_logger.warning` that includes `target`, `test_size`, `random_state`, and the repr of the exception. No behavior change to the fallback itself; only observability. Module now imports `logging` and defines `_logger = logging.getLogger(__name__)`.
- **M8 ‚Äî `FeatureValidationPipeline._high_corr_pairs` was O(n¬≤) Python with `.loc[]`.** The old nested loop did ~4.5M scalar `corr_matrix.loc[var1, var2]` lookups at 3000 features. Rewritten with `np.triu_indices(n, k=1)` to lift the upper triangle in one vectorized shot, `np.isfinite & (np.abs > threshold)` mask, then set-membership for `pair_type`. **Output is numerically identical** to the loop ‚Äî verified by the regression test `test_pipeline_medium_0400.py::TestM8HighCorrPairsVectorized::test_matches_loop_reference` which keeps a copy of the loop as a numerical oracle. No behavior change, just faster.
- **M9 ‚Äî `SampleAnalysisPipeline` silently spawned up to 960 splits by default.** Default `4 targets √ó 4 windows √ó 3 ratios √ó 20 seeds` grid √ó `SampleSplitter` per cell, no preview. Fix: new `dry_run: bool = False` config field short-circuits `run()` to skip all computation and only return an estimate dict in `split_candidate_summary`; new public method `estimate_split_count(data=None) -> dict` returns the grid product plus, when `data` is provided, `available_oot_periods` and `windows_usable`. Nothing in the default execution path changed ‚Äî the preview is strictly opt-in.

**Durable lessons across the batch:**
- **`or` on a container defaults to non-empty**: `cfg.get("x") or default` treats a deliberate `{}` / `[]` / `""` the same as "unset." When a user's intent might be "explicitly empty," write `is None` instead. Applies to M1, but grep every Pipeline config-consumer for the same idiom before it lands again.
- **A default value that requires a column is a footgun, not a feature.** `cross_vars=["rating"]` (M5) is a two-liner in the config file that quietly costs a KeyError to every user without the specific rating column the author had in mind. Any list-typed config default should be `[]` unless the column is one SMF itself creates.
- **`pd.to_numeric(..., errors="coerce")` followed by a truthiness check is a silent lossy conversion.** M6's fix (opt-in mode + per-column loss warn) is the pattern to reach for any time an object column might contain a mix of parseable numbers and legitimate string labels.
- **`np.triu_indices` is the vectorization for anything shaped like a pair-loop over a symmetric matrix** (M8). If a Pipeline method has `for i, ...: for j in ... [i+1:]: matrix.loc[i, j]`, it's a candidate.
- **A grid-search default that multiplies to hundreds of runs needs a `dry_run` or `estimate_*` counterpart on the same config** (M9). Adding it as a public method (not just a private helper) keeps the escalation ladder from having to re-derive the same math on the caller side.

## Defensive hardening batch (fixed 0.4.1)

Three independent LOW-severity gaps across three modules. All three took a code path that could silently ship wrong or misleading numbers and made it loud ‚Äî either raising an explicit error with an actionable message, or emitting a warning that names the model / column / count involved. No behavior change on healthy inputs.

- **L1 ‚Äî `RejectInferencePipeline._default_hard_cutoff` returned NaN on all-NaN scores.** The old fallback `return float(approved[cfg.score_col].median())` silently returned NaN when every bad-sample score was NaN and the overall score column was also all-NaN. That NaN threaded into every downstream `score < cutoff` comparison in `HardCutoffInferrer`, producing an empty inferred rejected set with no signal at all. Fix: check `np.isfinite(fallback)` and `raise ValueError` with a message pointing at `ri_method_params['hard_cutoff']['cutoff']` when the median itself is NaN; emit a `RuntimeWarning` (naming `cfg.score_col` and the median value) when only bad-sample scores are NaN but the overall median is finite. Healthy path untouched.
- **L2 ‚Äî `predict_positive` had no shape/length/finiteness contract.** The old code took `pred_arr[:, 1]` for 2-D `(N, ‚â•2)` and `pred_arr.reshape(-1)` for everything else. Silent failure modes: (1) 2-D `(N, 3+)` output from a stray model wrapper still went through `[:, 1]` with no warning; (2) any model whose `predict_proba` returned fewer rows than input (silent row drop inside `LRMaster` variants or custom wrappers) got broadcast into misaligned score columns downstream; (3) NaN/Inf predictions propagated silently. Fix: explicit branch for `ndim ‚àà {1, 2}` with 2-D `shape[1] ‚àà {1, 2}` accepted, everything else raises `ValueError`; explicit `len(result) == len(data)` assertion raises with the model class name on mismatch; NaN/Inf warn (RuntimeWarning) with count and model class. Sklearn `(N, 2)` and LRMaster 1-D paths both work as before, just with the assertions on top.
- **L3 ‚Äî `_ivks_from_binner` mixed `Interval + NaN` groupby was pandas-version-dependent.** `adapter.assign_bins()` can return a mix of `pd.Interval` objects and `NaN` for the missing bin. The old `groupby("bin", dropna=False)` used `sort=True` implicitly ‚Äî under pandas <2.0 that mixed-dtype key raised `TypeError: '<' not supported between instances of 'Interval' and 'float'`; under pandas ‚â•2.0 it worked but on an implementation detail. Fix: cast bins to `object` dtype, replace NaN with an explicit `"__MISSING__"` string sentinel via `np.where(pd.isna(bin_arr), "__MISSING__", bin_arr)`, and set `sort=False` on the `groupby`. Downstream `sort_values("bad_rate")` handles ordering anyway, so the reordering is invisible outside the intermediate frame. Numerically identical on healthy pandas ‚â•2.0.

**Durable lessons across the batch:**
- **Any `.median()` / `.mean()` / `.percentile()` used as a fallback needs a `np.isfinite` guard before returning.** Aggregation over all-NaN is a real thing (M1 style), and NaN propagating through downstream comparisons is invisible until the wrong result ships. The two-line pattern (`if not np.isfinite(x): raise ValueError(...)`) is cheap and belongs anywhere an aggregate is the escape hatch of a `len(...) == 0` branch.
- **Anywhere a model's `predict_proba` output becomes a DataFrame column, assert length equality with the input frame.** Row-drop-inside-a-model-wrapper is impossible to notice from the calling code because the model looks fine and the assignment doesn't error ‚Äî but the join between the wrong-length score array and the input frame silently reindexes/broadcasts. `assert len(result) == len(data)` is the cheapest way to catch it and names the actual failure.
- **`groupby(..., dropna=False)` on a mixed-dtype key relies on implementation detail.** If NaN is one of the possible key values and everything else is a non-comparable object type (Interval, custom class, mixed), replace NaN with a sentinel of the *dominant* dtype and pass `sort=False`. Costs nothing on the healthy path and removes an entire class of version-dependent failures.
- **"LOW severity" here means the code path is rarely hit, not that the failure is minor when it is hit.** Every one of these produced silently wrong data on trigger ‚Äî L1 empties an inferred set, L2 misaligns a score column, L3 masks an IV computation crash under old pandas. Ship them as defensive hygiene, not as "someday when there's time."

## HIGH hotfix batch (fixed 0.4.2)

Five independent HIGH-severity gaps found during a full re-read of 0.4.1 new code plus the non-Pipeline modules (`Core`, `Sample`, `Feature`, `Model`, `Eval`, `Explainability`). All five either produced silently-wrong outputs or corrupted persisted state without raising. Shipped together in 0.4.2 as a pure hardening release ‚Äî no new features, no API additions beyond one new opt-in policy field on `PSI_Tool`.

- **N11 ‚Äî `Core/ODPS_Tool.upload_df` was non-atomic: delete-then-recreate.** The old sequence was `o.delete_table(name, if_exists=True)` ‚Üí `o.write_table(name, df)`. If the process died, ODPS quota-blocked, or the network dropped between the two calls, the target table was left dropped with no data ‚Äî every reader of that table then failed with "table not found" until the next successful run. Downstream Pipeline monitoring jobs that read a rolling snapshot table were the observed victims. Fix: default `atomic=True` writes to a temp table `{name}__tmp_{uuid}` first, then `ALTER TABLE ... RENAME TO ...` swaps it in and drops the old one; on any exception mid-write the temp is cleaned up and the original table is left untouched. Legacy delete-then-recreate path preserved as `atomic=False` for callers who explicitly want the old behavior (rare ‚Äî mostly test fixtures). No signature change beyond the new kwarg.

- **N16 ‚Äî `Sample/Sample_Split.split_df` dropped `exclude_cols` from the output frames.** The old implementation ran `X = data.drop(columns=exclude_cols)` then split `X` ‚Äî the returned train/test frames were missing every column the user had asked to exclude *from splitting*, not from the output. Users passing `exclude_cols=["user_id", "loan_date"]` to keep those columns out of the stratification key lost them entirely from the downstream training frame, and the failure was silent until a later step tried to read `user_id` back. Fix: split on index (`np.arange(len(data))` + sklearn's `train_test_split` on indices with the stratify vector), then `data.iloc[train_idx]` / `data.iloc[test_idx]` ‚Äî both output frames retain **every** column including `exclude_cols`. `exclude_cols` now only governs what participates in the split criteria, matching the user-facing contract in the docstring.

- **N17 ‚Äî `Sample/Reject_Infer.HardCutoffInferrer.infer` propagated NaN scores as inferred labels.** The old `rejected_scored["target"] = (rejected_scored[score_col] < cutoff).astype(int)` treated NaN < cutoff as False and assigned `target=0` to every reject with a missing score. In the observed failure, a stray model wrapper dropped ~4% of reject rows to NaN scores; those 4% then joined the training frame as confidently-good synthetic labels, biasing the next-cycle model toward approving them. Fix: `np.isnan(rejected_scored[score_col]).all()` raises `ValueError` with a message pointing at the prescore column; partial NaN emits a `RuntimeWarning` (naming `score_col` and the NaN count) and assigns `target=NaN` (float dtype) to those rows so downstream `.dropna(subset=["target"])` filters them out cleanly instead of silently mislabeling them. Added `import warnings` at the top of the module.

- **N23 ‚Äî `Feature/Weighted_Screen._weighted_bin_distribution` force-unioned a missing bin sentinel and clipped negative weight totals.** The old code took every fitted bin plus a hard-coded `_MISSING_BIN` string and computed `np.clip(bin_weight_sum, 0, None)` on the sample-weight aggregate. Two failure modes: (1) when the input frame had no missing values, the returned distribution still contained a zero-count `_MISSING_BIN` row, inflating the bin count reported by every downstream PSI/IV consumer by exactly one and shifting the distribution's normalization; (2) the `clip(..., 0, None)` silently absorbed negative-weight sums (a real error signal for wrongly-signed IPTW weights) instead of raising. Fix: `_weighted_bin_distribution` returns observed-only bins ‚Äî no `_MISSING_BIN` union unless a NaN is actually present in the source column ‚Äî and drops the `clip` entirely. Negative weight sums now propagate as-is; the caller's downstream `assert (weights >= 0).all()` at the entry to `weighted_feature_screen` (added in 0.3.7, retained) catches them at the boundary with a clearer message. `content` kwarg kept for signature stability even though it no longer branches internal logic.

- **N29 ‚Äî `Feature/PSI_Tool._calculate_single_psi` silently dropped the missing bin from both distributions.** The old implementation ran `.dropna()` on both the expected and actual bin-share Series before the PSI formula. Consequence: any drift in the **missingness rate itself** was invisible. A production example from the 0.4.1 audit: expected 5% missing, actual 30% missing, all observed bins otherwise stable ‚Üí old PSI = 0.007 ("no drift"), correct PSI = 0.53 ("severe drift"). Fix: new config parameter `missing_policy: Literal["drop", "include", "warn_and_drop"] = "drop"` on `PSI_Tool.__init__` and `calculate_psi` ‚Äî **default kept at `"drop"` for exact numeric backward-compat** with 0.4.1 and earlier. `"include"` keeps the `__MISSING__` bin in both distributions and includes its share in the PSI sum (the correct behavior). `"warn_and_drop"` matches old numeric behavior but emits a `UserWarning` naming the column and the difference in missing rates when either side has non-zero missing share. Module now defines `_MISSING_BIN = "__MISSING__"` as a top-level constant shared with `Weighted_Screen` for consistency. **The next minor release will flip the default to `"include"`** ‚Äî noted in `changes_042.md` ‚Äî following the same 0.3.19‚Üí0.4.0 pattern that flipped `ScoreComparisonPipelineConfig.cross_vars`.

**Durable lessons across the batch:**
- **A two-step "delete then recreate" against remote state is not atomic and needs an explicit temp-swap or transactional wrapper** (N11). Grep for `.delete_table(...)` / `.drop_table(...)` immediately followed by a `.write_...` / `.create_...` in every Core/IO module ‚Äî that pattern is unsafe by default. `atomic_swap_table` / `atomic_swap_partition` are the utilities to reach for.
- **A column marked "exclude from computation X" must not disappear from the output of X** (N16). The user-facing name of the parameter tells you the contract; if the implementation ends up doing more than that contract says, the extra behavior is a silent bug even if the tests pass.
- **NaN comparisons default to False, so `(col < threshold).astype(int)` silently assigns 0 to every NaN row** (N17). Any boolean-cast-to-int of a comparison involving a column that might be NaN needs either an explicit `.notna()` mask or a NaN-target assignment; the "which one?" depends on whether the caller wants the rows silently included as False or filtered out downstream ‚Äî but "silently False" is virtually never the right answer for a scored population.
- **`np.clip(..., 0, None)` on an aggregate that should never be negative silently absorbs the error signal** (N23). If the value going into `clip` is impossible-under-correct-input, clip is not defensive ‚Äî it's a bug-hider. Prefer an assertion at the boundary where the invariant is guaranteed (the entry to `weighted_feature_screen`) and let the internal aggregate propagate whatever the math produces.
- **`.dropna()` on a distribution before a divergence metric silently discards the missingness dimension of the drift** (N29). Missing-rate drift is real drift; the tooling either has to include the missing bin in the metric or has to warn the caller that it's being dropped. Never silently drop.
- **When a fix changes numbers, ship it behind a flag with the old default, document the intended flip, and flip on the next minor release** ‚Äî the 0.3.19‚Üí0.4.0 `cross_vars` pattern is now the standard for any behavior-changing default in this codebase. `missing_policy="drop"` is 0.4.2's version of the same pattern.

## MEDIUM batch (fixed 0.5.0)

Two MEDIUM-severity items shipped together in 0.5.0 as a coordinated correctness release. Both change observable numeric/logging behavior relative to 0.4.x; both are noted in `docs/changelog/v0.5.0.md` with migration guidance.

- **N29 ‚Äî `missing_policy="drop"` default was left in place across the 0.4.x line and was still hiding missing-rate drift.** In 0.4.2 the `missing_policy` kwarg was introduced on `PSI_Tool.__init__` / `calculate_psi` with the intended flip to `"include"` on the next minor release, following the 0.3.19‚Üí0.4.0 `cross_vars` pattern. 0.5.0 completes that flip and simultaneously widens the API surface: the kwarg is now accepted (and forwarded internally) on **seven surfaces** ‚Äî module-level `calculate_psi`, `calculate_psi_over_time`, `psi_report`, `plot_psi_trend`, `compare_distributions`, plus `PSICalculator.__init__` and `PSICalculator.calculate`. All seven default to `"include"`. Internal helper `_calculate_single_psi` default was also flipped from `"drop"` to `"include"` so any caller that reached the helper directly gets the corrected behavior too. Concrete numeric impact on the audit fixture (5% expected NaN ‚Üí 30% actual NaN, all observed bins otherwise stable): PSI ‚âà **0.0009 under the old default ‚Üí 0.525 under 0.5.0**. Callers that need bit-exact continuity with 0.4.x reporting must pass `missing_policy="drop"` explicitly at every call site ‚Äî the module-level constant `_MISSING_BIN = "__MISSING__"` continues to name the synthetic bin used by the include path. Sample-weighted PSI (`Weighted_Screen`) already used the include-style path since 0.3.14 and is unchanged; only unweighted `PSI_Tool` surfaces flip.

- **N38 ‚Äî `WOE_Monotone_Binner.apply_woe` silently mapped unseen categorical levels to the training-time missing-value WOE.** The old code, on encountering a category present in the transform frame but absent from the fitted `value_map`, fell through to the same branch as `NaN` and assigned the fitted `nan_woe`. Two independent failure modes: (1) rare-category drift (a value that appeared only in OOT or in production) received a WOE that was fitted for a **structurally different** population ‚Äî the missing subgroup ‚Äî with no signal in the logs; (2) users who had already routed missing values through `unseen_category_policy` semantics in adjacent tools (e.g. sample-weighted screening) had no way to force an error or a distinguishable output for genuinely-new levels. Fix: `apply_woe` gained `unseen_category_policy: Literal["warn", "error", "silent"] = "warn"`. On any transform call where a categorical column contains a level not in the fitted `value_map`, the binner (a) records a per-column entry in `self._unseen_category_stats` ‚Äî a `Dict[str, dict]` reset to `{}` at every `apply_woe` invocation, populated with `{"unseen_levels": [...], "unseen_count": int, "unseen_share": float}` ‚Äî and (b) either emits a `RuntimeWarning` naming the column, levels, count and share (`"warn"`, default), raises `ValueError` with the same details (`"error"`), or does nothing user-visible while still populating the stats dict (`"silent"`). The mapped WOE value for unseen levels is still `nan_woe` ‚Äî the fix is *observability + opt-in-strictness*, not a new imputation. Only the categorical branch (`vr.get("is_categorical") is True`) runs the detection; numeric bins are unaffected because unseen numeric values already fall into an existing bin range or trigger the missing bin explicitly. `warnings` was already imported at module top so no new import.

**Durable lessons across the batch:**
- **A shipped-with-old-default flag with an announced flip must actually flip on the next minor release** (N29). Leaving `missing_policy="drop"` in place across two minor bumps would have quietly re-established the original silently-wrong default as the codebase's steady state. The `changes_042.md` line "the next minor release will flip the default" is now a promise the codebase kept; the pattern only works if the flip happens.
- **Every module-level convenience wrapper around a class-based API is a place the kwarg has to be re-plumbed** (N29 API surface expansion). `PSI_Tool` shipped `missing_policy` on `PSICalculator.__init__` in 0.4.2 but the six module-level functions (`calculate_psi`, `calculate_psi_over_time`, `psi_report`, `plot_psi_trend`, `compare_distributions`, and internally `_calculate_single_psi`) did not accept or forward it. Users reaching for the module-level API ‚Äî the documented entry point ‚Äî got the old default even in 0.4.2. Whenever a new kwarg is added to a class method, grep the same module for `def ` at column 0 and audit whether each top-level function needs the same kwarg forwarded.
- **Silently mapping unseen categorical levels onto the missing-value bucket conflates two distinct populations** (N38). "Missing at training time" and "present in production but never seen at training time" are different subgroups with different risk profiles. Any WOE-style transform must expose *at minimum* an observability hook (the `_unseen_category_stats` dict) even if the default action stays soft (warn). Users who care can escalate to `"error"`; users who don't can still read the stats dict downstream and log a monitoring metric.
- **`-W error::RuntimeWarning` in a user's pytest config will now break `apply_woe` on any dataset with unseen categoricals** (N38 downstream trap). This is the single most likely 0.5.0 upgrade break for engineering teams who treat all warnings as errors. The migration answer is `unseen_category_policy="silent"` at the call site (or `warnings.catch_warnings()` around the call); both preserve the stats dict for offline monitoring. Documented in `v0.5.0.md` migration section.
- **PSI numeric values are not comparable across the 0.4.x ‚Üí 0.5.0 boundary on any dataset with drifting missingness** (N29 downstream trap). Historical PSI reports stored from 0.4.x will look artificially calm compared to fresh 0.5.0 runs on the same data; teams that alert on absolute PSI thresholds (e.g. `psi > 0.25`) need to either re-baseline against a 0.5.0 recomputation or pass `missing_policy="drop"` explicitly to freeze the old numbers for continuity. Recommended: re-baseline. The old numbers were wrong.

## Feature / WOE hygiene batch (fixed 0.5.1)

Eight medium-severity hygiene issues shipped together in 0.5.1. Most were not crashers; they were silent numeric distortions or "where did my variable go?" diagnostics gaps.

- **Weighted IV zero-mass bins** used to let zero bad/good mass bins produce `inf` / `nan`, and the synthetic missing bucket could inflate predictive IV. Fixed by excluding the missing bucket by default and skipping class-degenerate bins with a warning.
- **`_assign_bins(edges=None)`** used to map every non-null value to `"all"` and produce IV=0. Fixed by raising a clear `ValueError` by default; `on_null_edges="warn_and_zero"` preserves the old permissive behavior explicitly.
- **`Feature_Screen` after missing-rate drop** used to refit later stages on the original feature list in some paths. Fixed so WOE / PSI / IV only receive the current surviving feature list.
- **PSI one-sided buckets** used to floor the absent side to `1e-6`, creating huge PSI spikes for small new-bucket mass. Fixed with `psi_missing_bucket_policy="smooth_laplace"` by default; `"floor_1e6"` is the old report-compatible path and `"exclude"` removes one-sided buckets from the sum.
- **Feature insight failures** in `Feature_Insights` and the WOE-engine patch used to disappear through broad or silent exception handling. Fixed by exposing `failed_variables` and emitting a single warning for expected per-variable failures; unexpected exceptions should propagate.
- **`WOE_Master` missing sentinel** default changed from `-999999` to `SMF_MISSING_BIN`. Passing finite sentinels now warns, and real collision in training data raises.
- **Empty WOE mapping tables** now raise a contextual `WOE_Master.woe_dict is empty` error instead of raw `No objects to concatenate`.

**Durable lesson:** feature-screening tools need loud failure surfaces. "Variable disappeared" and "metric is exactly zero" are often swallowed upstream exceptions, not clean statistical conclusions. Check `failed_variables`, sentinel collisions, and PSI bucket policy before trusting a calm report.

## Test-environment baseline (post-0.5.0)

Not a code bug ‚Äî a **coverage-hiding gap** discovered while validating 0.5.0 in a fresh sandbox venv. Recorded here because the failure mode is silent (green run, 22 SKIPPED) and easy to miss on future releases.

- **`pytest --tb=short -q` reporting `N passed, 22 skipped, 0 failed` on a fresh SMF pytest venv is the default state, not a healthy state.** The suite covers three optional SMF capabilities ‚Äî ODPS I/O (`odps` namespace), SHAP explainability, LIME explainability ‚Äî that the main wheel deliberately does **not** declare as runtime dependencies. **Durable lesson:**
- **A green pytest run with non-zero SKIPPED count is a monitoring failure, not a success.** Every SKIP has to trace to either a documented `@pytest.mark.skip(reason=...)` or a missing optional import; the second category is a **coverage gap**, not an environment quirk. Treat any new `SKIPPED [could not import 'X']` line during release verification as a blocking finding and either (a) add `X` to `requirements-dev.txt` or (b) mark the test with an explicit reason that survives grep. 0-skip is the correctness baseline for this codebase from 0.5.0 forward.

Fill values are computed **on the training frame only** (¬±inf treated as missing; all-NaN columns fall back to 0.0), applied to train/val with a loud `UserWarning`, and ride with the model via a module-level `_LRNanFillWrapper` that re-applies them at every predict call ‚Äî train and scoring can never drift onto different imputations. The wrapper blocks dunder delegation so pickle serializes it (not the inner LRMaster's `__getstate__`), and `_save_pipeline_model` persists `lr_nan_handling` + `lr_fill_values` in the artifact metadata. **Durable lessons:**
- When a fix lets a previously-crashing input travel further down a staged pipeline, audit every later stage for the same assumption ‚Äî "fixed at IV" would have just relocated the crash to corr.
- Any imputation applied at fit time must be captured as state and re-applied at predict time; filling only the training frame is a train/serve skew bug wearing a fix's clothes.
- A module-level blanket `filterwarnings("ignore")` anywhere in a library silently disables every carefully-designed warning everywhere else in it.

## Silent equal_freq WOE plot failure (fixed 0.6.2)

`FeatureValidationPipeline(woe_engine="equal_freq", plot_outputs=True)` silently produced **zero PNGs** under `figs/woe/` on every 0.5.x / 0.6.0 / 0.6.1 release, with no log line and no exception surfaced to the user. Two independent defects layered on top of each other:

- **API-surface defect (WOE_Master)**: `WOE_Master.plot_bivar_graph(self, data, group, dirname, varlist=None)` declared `group` as a **required positional argument**, even though the underlying primitive `get_bivar_graph(..., group=None, ...)` has always accepted an ungrouped call. `feature_validation._plot_woe` invoked it as `engine.plot_bivar_graph(transformed, dirname=str(base_dir))` ‚Äî no `group` ‚Äî for the base (ungrouped) figure, and got a `TypeError: missing 1 required positional argument: 'group'` on the very first call.
- **Silencer defect (feature_validation)**: the entire `_plot_woe` body was `try: ... except Exception: return`. The `TypeError` above was swallowed with **no log, no warning, no re-raise**. The monotone engine uses a different code path (`engine.plot_woe_graph`), so the blind spot only manifested on `equal_freq` and any other engine whose plot entry point is `plot_bivar_graph`. It survived from 0.5.x through 0.6.1 because the existing plot tests only covered the monotone branch and the equal_freq branch through `CreditModelPipeline`, whose call site correctly passed `group=` as a kwarg.
- **Secondary directory defect surfaced during the fix**: `WOE_Master.plot_bivar_graph` did not `mkdir` its composed `save_dir`. Callers were expected to pre-create every subdirectory; `feature_validation._plot_woe` only created the base `figs/woe/<target>`, so even after the `TypeError` was fixed the first PNG write would `FileNotFoundError`. Fix moved `os.makedirs(save_dir, exist_ok=True)` into `plot_bivar_graph` itself ‚Äî the method that composes the path also owns creating it.

Fix (0.6.2):
- `WOE_Master.plot_bivar_graph` ‚Äî `group` defaults to `None`; positional order preserved (`data, group, dirname, varlist`) for backward compatibility; `dirname=None` now raises a clear `TypeError` at call time; the method owns its `mkdir`.
- `feature_validation._plot_woe` ‚Äî passes `group=None` explicitly at the ungrouped call site; the outer `except Exception:` block now emits a `_logger.warning` naming engine, target, output_dir, and the underlying exception, then returns. Plot failure still does not kill the pipeline, but it is no longer invisible.
- New regression coverage (`SuperModelingFactory_pytest/test_pipeline_bugfix_062.py`): API-surface (TestN45), end-to-end equal_freq PNG output through FeatureValidationPipeline (TestN46), and a logger-warning assertion for engine-side plot failures (TestN47).

**Durable lessons:**
- **A bare `except Exception: return` in a library is a bug in itself**, independent of any specific defect it hides. Every silenced exception is a monitoring failure waiting to happen. The canonical pattern in SMF from 0.6.2 forward: `except Exception as exc: _logger.warning("...error=%r", ..., exc); return` ‚Äî log first, decide the recovery second. Silent failure is *more* dangerous than a crash because a crash gets fixed by the next release; a silent failure accumulates for years.
- **When you add coverage for a new configuration surface (like `plot_outputs`), sweep every engine/adapter path that flows through the same output block.** The 0.6.1 `plot_outputs` toggle was fully tested on the monotone engine and never exercised the equal_freq branch end-to-end ‚Äî because the equal_freq path had been silently broken all along, no baseline existed to compare against. Grep for every branch of every `hasattr` / `get_engine_name()` switch and put at least one end-to-end assertion behind each.
- **Any method that composes a path should own creating it** (`os.makedirs(..., exist_ok=True)`). Splitting the responsibility between caller and callee (caller creates base, callee expects subdirs to exist) is a bug pattern that survives every refactor because it only fails on the third-least-common call site.

## equal_freq WOE plot close-out (fixed 0.6.3)

0.6.2 only fixed **half** of the silent-plot problem above. The 0.6.2 diagnosis included this sentence, which turned out to be wrong:

> "The underlying `get_bivar_graph` primitive has always accepted `group=None`."

It hadn't. The signature default was there, but the function body's `if group:` guard sat **after** the crash point. `get_bivar_graph` did the reference-WOE plot loop first (which writes PNGs successfully), then unconditionally called `get_mapped_woe_summary(..., grp_name=[group])` and `align_bin_num(..., grp_name=[group])` ‚Äî those internally do `data.groupby([None])`, which raises `TypeError: 'NoneType' object is not callable`. Only after that unreachable code did `if group:` gate the plotting loop.

Net effect on 0.6.2 in the field:

| engine       | base PNGs                              | `by_<group>/*.png` | plot exception          |
|--------------|----------------------------------------|--------------------|-------------------------|
| monotone     | 6                                      | 3                  | none                    |
| equal_freq   | 3 (ref-plot loop runs before ‚ë° crashes) | **0**              | **TypeError** every run |

Because the 0.6.2 pipeline `try` block caught the `TypeError` and `return`ed, the subsequent `for group in cfg.woe_plot_groups:` loop that emits `by_<group>/*.png` never ran. The 0.6.2 `logger.warning` fix made the residual crash visible ‚Äî that's how the second half was actually caught.

Fix (0.6.3): hoist `if group:` in `WOE_Plot_Tool.get_bivar_graph` so it wraps the `summary + align + plot` triplet. Docstring in `WOE_Master.plot_bivar_graph` corrected to remove the 0.6.2 wrong-assumption sentence.

Regression coverage (`SuperModelingFactory_pytest/test_pipeline_bugfix_063.py`): TestN46b (base plots emit no `WOE plot generation failed` warning), TestN48 (equal_freq + `woe_plot_groups=['chan']` produces `by_chan/*.png`), TestN49 (white-box: `get_bivar_graph(group=None)` does not raise).

**Durable lessons (three, all sharing one root cause):**

- **Signature defaults are not proof the body handles the default value.** Both the 0.6.1 and 0.6.2 wrong-turns came from reading `def get_bivar_graph(..., group=None, ...)` and stopping there. When the diagnosis hinges on "the underlying primitive supports X", read the body to the point where X is actually consumed ‚Äî or write a white-box test that pins the claim.
- **"Output exists" ‚âÝ "no error occurred."** The 0.6.2 D5 test asserted "PNG count > 0" and passed on 0.6.2 while every run also raised ‚Äî because loop ‚ëÝ wrote PNGs *before* loop ‚ë° crashed. Any test path where the failure is caught by a surrounding `try/except` must assert **both** (a) the presence of the desired artifact and (b) the absence of warnings/exceptions from the operation. This is now `TestN46b`'s job: PNGs exist AND `caplog` shows no `"WOE plot generation failed"` warning.
- **The 0.6.2 `logger.warning` justified itself on day one.** It was the visibility half of a two-part fix; even though the behavioral half was wrong, the visibility half was what surfaced the residual crash for 0.6.3. When a bug has a visibility layer and a behavioral layer, always ship the visibility layer even if you're not 100% sure the behavioral layer is complete ‚Äî you'll find out from the next crash.

## OOT governance gaps (found in the 0.6.4 audit, fixed after 0.6.4)

The 2026-07-16 defect review (G10/G11/G12, P0) confirmed three OOT-leakage paths in 0.6.4; all three closed together in the next release. **Resolved defaults stay legacy in 0.6.x ‚Äî the announced next-minor flip inverts them.**

- **G10 ‚Äî missing OOT was silently synthesized from OOS.** Four sites (`credit_model._split_data` √ó2, `feature_validation._split_data` √ó2) ran `oot = oos.copy()` with no warning, so "OOT" metrics on OOT-less runs were OOS metrics wearing an OOT label. Fix: shared `Pipeline/_common.resolve_missing_oot` ‚Äî legacy synthesis now warns loudly ("stand-in OOT"); `synthesize_missing_oot=False` keeps OOT out entirely (CMP: missing `"oot"` key; FVP: empty frame ‚Äî every downstream consumer already guards with `len(oot)`). `CreditModelPipelineResult.split_governance` + saved model metadata record `oot_synthesized` / `oot_withheld` / `candidate_mode`.
- **G11 ‚Äî hyperparameter search hard-coded OOT.** Both `_run_lr_search` AND `_run_optuna` (the audit doc only named LR; optuna had the identical pattern) built `eval_sets={"oos", "oot"}` with default `objective="oot_gap_penalized"` ‚Äî every candidate parameter set was scored on real OOT. Fix: `search_eval_splits` (None = legacy oos+oot, degrading to available splits; explicit list validated strictly), `search_objective_when_no_oot` fallback (`max_primary`), and validation that an explicit `oot_gap_penalized` without an available gap-ref split raises instead of silently degrading. `optuna_params["common"]["eval_sets"]` overrides still work but every key passes the hard gate.
- **G12 ‚Äî backward consumed OOT via `test_data_dict`.** `{"oot": splits["oot"]}` fed `Backward_Tool.datain_all`, so variable elimination could watch OOT metrics round by round. Fix: `backward_validation_split` (default oos) + `backward_report_splits` (None = legacy ["oot"], flip to [] next minor); `backward_params["init"]["test_data_dict"]` keys pass the hard gate.
- **`candidate_mode=True`** is the one-switch preset: forbids oot everywhere, never synthesizes, evaluation ins/oos only, search oos only, backward reports nothing; real OOT rows in the data are split out but never consumed (`oot_withheld=True`). Explicitly contradicting settings raise (no silent override). `forbidden_splits` is the underlying hard gate ‚Äî any component (including optuna/backward param-dict smuggling) requesting a forbidden split gets a `ValueError` naming the component.

**Durable lessons:**
- A silently synthesized placeholder that shares a real artifact's name ("oot") will eventually be accepted as the real thing ‚Äî placeholders must be loud, opt-out-able, and recorded in the run's metadata.
- When one search path is audited (LR), grep for the sibling paths (optuna/GBM) before declaring the leak closed ‚Äî hard-coded `eval_sets` existed in both.
- Governance switches need a hard gate at the consumption site, not just resolved config defaults ‚Äî otherwise param-dict escape hatches (`optuna_params`, `backward_params`) silently reopen the hole.

## Evaluation governance gaps (found in the 0.6.4 audit, fixed after 0.6.5)

The remaining P0 items from the 2026-07-16 defect review (G13/G14/G15/G16), closed together after 0.6.5. All new config defaults preserve legacy numbers; the announced next-minor flip covers `gains_ascending` ‚Üí True.

- **G13 ‚Äî CMP could only evaluate one label.** The underlying `PerformanceEvaluator` had grown native multi-label support (`tgt_name` accepts a list: per-label evaluation, vertical concat with a `tgt_name` column, score reuse), but `_evaluate_models` hard-passed the single `cfg.target_col`. Fix: `eval_target_cols` ‚Äî effective label set `[target_col] + eval_target_cols`; the frozen `pred_{model}` score is computed once per dataset and reused across labels (regression-asserted via a predict-call counter). Single-label output shape is byte-identical.
- **G14 ‚Äî no "all raw features missing ‚Üí score=-1".** `Core.scoring()` already had `all_missing_spec_value` with the `pd.isnull(...).sum(axis=1) == len(varlist)` test; the pipeline evaluation chain had nothing. Fix: `all_missing_score_value` + shared `Pipeline/_common.all_missing_mask` (same membership test, so pipeline evaluation and production scoring agree row-for-row ‚Äî parity-tested against `Core.scoring` on a raw-source GBM). The override is defined on **raw** features (woe-source models map back via `_woe_to_raw_features`) and rides in the artifact metadata (`all_missing_score_value`, `all_missing_raw_features`).
- **G15 ‚Äî sentinel scores were quantile-binned like real scores.** Unweighted internals already accepted `spec_values`; `PerformanceEvaluator.__init__` didn't expose it and the weighted path had no special-value concept at all. Fix: evaluator-level `spec_values`/`ascending` params threaded through; `weighted_eval_utils.get_gains_table`/`dataset_summary`/`get_perf_summary` gained `spec_values` (special rows split out before `rank_bins`, reported as `special:<value>` gains rows with ranking columns NaN; summary AUC/KS/avgScore computed on non-special rows, `N_SPECIAL`/`N_SPECIAL_RAW` reported); unweighted AUC/KS/figures pre-filter special rows.
- **G16 ‚Äî weighted gains direction was inverted and eval weights were welded to training weights.** `rank_bins` sorted by `-score` unconditionally (bin 1 = high score) while parts of the unweighted path plot ascending; CMP had no `ascending` control and hard-passed `cfg.weight_col` as `eval_weight_col` into search and evaluation. Fix: `rank_bins(..., ascending=False)` param; `gains_ascending` (None = every path keeps its own legacy default; explicit bool threads through summary/gains/weighted uniformly; **flips to True next minor**); `eval_weight_col` (`"inherit"` default = legacy; `None` = evaluate unweighted even when training is weighted; applies to `_evaluate_models`, lr_search/optuna `eval_weight_col`, backward `validation_weight_col`).
- **Known limitation:** `gains_ascending` does not yet control `evaluate_model.evaluate_performance`'s distribution figures (fixed-signature legacy plotting function, no `**kwargs`); special rows ARE pre-filtered from those figures. Slated for the flip release.

**Durable lessons:**
- Before building a "missing capability", introspect the layer below ‚Äî two of four P0 gaps (multi-label evaluate, all-missing override) already existed underneath and only needed wiring.
- A sentinel value that enters a ranking metric is not neutral: -1 in an AUC or a quantile edge quietly reshapes every bin. Sentinels need first-class routing (own bin, excluded from edges and ranking), not documentation.
- When two implementations of the same metric (weighted/unweighted) evolve separately, their conventions WILL diverge ‚Äî the weighted path inverted bin order for years because nothing asserted cross-path direction consistency.

## Weighted-path audit closure (found in CDC field use after 0.6.7, fixed 0.6.8)

The third and (for the screening layer) final round of weighted-vs-unweighted semantic divergence, after G16 (0.6.6, training weights leaking into evaluation) and the weighted screening bugs of 0.3.7‚Üí0.3.14. Field trigger: an FT-weighted CDC run where `iv_upper_threshold` failed to drop a hard leak feature that the unweighted monotone-engine run caught at IV‚âà24.

- **G02 upper band missed hard leaks on 3 of 4 dispatch paths.** Not a gate-wiring bug ‚Äî all four paths called `_iv_band_keep` ‚Äî but an IV-basis bug: the weighted IV's `iv_guard` excludes class-degenerate bins from the sum, and a hard leak makes bins class-PURE, so the strongest possible signal contributed *nothing* (measured 0.6‚Äì1.7 for a true-IV‚âà12 leak; fully-pure features measured 0.0). The legacy gains-table path zeroes pure bins the same way via `filliv`. Only monotone-engine report IV (zero-cell floored) was immune. **The 0.6.7 batch's own G02 tests were green because their `f_leak = y*2 + N(0,0.55)` is a SOFT leak ‚Äî distributions overlap, no bin is pure, the degenerate regime was never exercised.** Fix: `_weighted_iv_detail` computes a zero-cell-FLOORED IV (shares floored at `content`, the PSI floor) in the same pass; every path gates the upper band on it; drops record `metric=iv_floored`. Reported IV columns stay legacy; `upper=None` computes nothing.
- **G03/G04 evidence chain ignored `weight_col` entirely.** Per-group IVs, per-target IVs, point-biserial directions ‚Äî all unweighted even on weighted runs, so group-stability and multi-target decisions used a basis that does not match the modeling weights. Weights can flip every one of the three gate checks (verified constructions: monthly IV 4.6‚Üí0.2, per-target IV 2.2‚Üí0.01, direction +1‚Üí‚àí1 in 2 of 3 months). Fix: `_evidence_iv` + `point_biserial_direction` gained `sample_weight`; `_build_selection_evidence` resolves `weight_col` once on INS (AFTER the gates-off early return, so gate-less weighted runs stay byte-identical) and re-slices positionally per group/target. `n` stays a RAW row count (weights add no statistical sufficiency); snapshot records `evidence_weight_col`.
- **Corrected diagnosis from the 0.6.7 commit message:** the G03/G04 evidence IV never had the `iv_guard` pure-bin hole ‚Äî it uses the `+1e-6` additively-smoothed report formula where pure bins contribute large finite terms. The `iv_guard` hole's real landing spot inside the gates was `gate_iv_map` feeding **G05 truncation ranking and the G06 VIF tie-break**: legit features with class-pure bins ranked artificially low (guarded 0.2‚Äì0.5 vs floored ‚âà4) and got truncated in favor of weaker features. Fix: `_gate_ranking_iv_map` substitutes finite floored IVs for ranking ONLY when the upper band is armed (leaks already dropped by the band); unarmed keeps each path's legacy basis verbatim ‚Äî deliberately, because floored ranking without the band would rank an uncaught leak first.

**Durable lessons:**
- Degenerate-bin exclusion inverts exactly for leak detection: a guard built to keep noise out of a LOWER informativeness bound silently zeroes the strongest features at the UPPER bound. Any IV consumer must pick its basis per purpose ‚Äî guarded for floors, floored for leak ceilings, and ranking follows whichever hazard is already handled.
- Leak-detection tests need HARD leaks (class-pure bins, e.g. 80% separated mass + 20% mixed remainder so the suppressed IV lands innocently inside the band) ‚Äî soft leaks with overlapping distributions never enter the degenerate regime and prove nothing about it.
- "Does this stage honor `weight_col`?" must be asked of every evidence/diagnostic computation SEPARATELY from the main stage math ‚Äî this is the third time a side channel (evaluation weights, band diagnostics, gate evidence) silently stayed unweighted after the main path was weighted. The unit-weight parity test (`w=1.0` frame-equal to unweighted, bit-exact via bincount identity) is the cheap permanent guard; write it whenever a weight parameter is added.
- Weighted-vs-unweighted flip fixtures want frequency-scale weight contrast (1.0 vs 1e-3) on engineered signal rows, not random weights ‚Äî each check (IV floor, IV range, direction consistency) gets its own weight column so one frame calibrates all flips deterministically.
- When a fix's ranking/ordering side effect could resurrect a hazard the legacy behavior accidentally suppressed (guarded ranking buried leaks at the bottom), condition the new behavior on the hazard's dedicated control being armed instead of changing the default.
