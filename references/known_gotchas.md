# Known gotchas â€” a history log, not a current-state list

Every entry is tagged with when it was true. **Before repeating one of these to the user, check whether the installed version is at or past the fix** (introspect the relevant source â€” `inspect.getsource`) and say so either way: "this was real before v0.3.X, confirmed fixed in your current source" is a better answer than silently re-warning about something already closed, and better than silently assuming it's fixed without checking.

## Reject inference direction bugs (fixed 0.3.3)

Root cause across all of these: `Sample.Reject_Infer`'s inferrers were written assuming "higher score = lower risk," while `RejectInferencePipeline` fed them `prescore_prob` = P(bad) â€” the opposite polarity â€” with no parameter to reconcile the two.

- **HardCutoffInferrer** â€” `target = (score < cutoff)` under P(bad) input flags the *safest* rejects as bad and vice versa. Symptom that gave it away: inferred reject bad-rate came out *lower* than the real approved bad-rate, which makes no business sense (rejects should look worse, not better).
- **FuzzyAugmentInferrer** â€” rejected pseudo-label was `1 - score` (wrong direction, and continuous rather than a proper good/bad split); approved-row reweighting formula additionally suppressed toward zero any real row the prescore had misjudged, amplifying the prescore's own bias instead of correcting for it.
- **ParcelingInferrer** â€” approved population binned via `qcut` (equal-frequency), rejected population binned via `cut` (equal-width, over the *rejected* population's own score range) â€” two unrelated bin systems mapped onto each other by index, effectively assigning near-random bad rates.
- **Pipeline-level (`_train_and_evaluate_models`)** â€” `split_col` mode set `val = approved.copy()`, i.e. validation was literally the training population; every non-weighted method's `train_AUC` and `validation_AUC` came out identical to 6 decimal places, and GBM early stopping had no real holdout to stop against.
- **Continuous pseudo-labels got hard-binarized at `>0.5`** downstream in the pipeline, compounding whichever of the above was already wrong.
- **SimpleAugmentInferrer** hardcoded `np.random.seed(42)`, polluting the global RNG state and ignoring `cfg.random_state` (lower severity â€” the method is "assign the approved bad-rate at random" by design, so the direction bugs above don't apply to it, but reproducibility did).

**Fix:** `score_direction` parameter added to every inferrer + `ri_score_direction` (`"high_bad"`/`"high_good"`) on the pipeline config; `_default_hard_cutoff`'s percentile flips with direction; fuzzy rewritten to the textbook form (approved weight 1, rejected duplicated into weighted good/bad rows); parceling given shared `parcel_edges_`; `ri_validation_frac` added for a genuine holdout; SimpleAugment moved to an instance RNG.

**Durable lesson:** any score-consuming module needs its polarity convention pinned and checked against what's actually fed in. These bugs threw no exceptions â€” they just produced confidently-wrong, business-plausible-looking numbers (a "hard cutoff" method assigning rejects a *lower* bad rate reads as surprising but not obviously broken until you check the arithmetic).

## Weighted screening bugs (introduced 0.3.7, fixed 0.3.14)

- **Correlation silently blind to NaNs.** `_weighted_pearson_corr_matrix` ran `np.cov(..., aweights=w)` directly on raw (possibly-NaN) columns; any NaN anywhere in a column propagates `cov` to NaN, which then got `np.nan_to_num(nan=0.0)`'d â€” i.e. any feature pair where either column has missing values reads as *uncorrelated*, and both survive correlation dedup untouched. On this project's real feature matrices (20â€“35% NaN from cleaned sentinel values) this made the correlation stage close to a no-op for most pairs. Confirmed empirically with a synthetic 30%-NaN pair at true correlation â‰ˆ0.999: both columns survived pre-fix. The weighted path also ignored `corr_use_woe_bins` (the unweighted path respected it).
- **Silent "everything survives" fallback.** Every stage (PSI/IV/corr) used `current = keep or current` â€” if a threshold eliminated every remaining variable, the function quietly returned the untouched input set with no warning and no trace in the summary table.

**Fix (0.3.14):** `corr_nan_policy` (`"pairwise"` default â€” complete-case overlap per pair; `"median_fill"`; `"raise"`); weighted corr now also respects `corr_use_woe_bins`; `on_empty_stage` (`"keep_all_warn"` default â€” same fallback behavior, but now `warnings.warn`s and appends a `*_fallback` row to `.summary`; `"raise"` for strict use).

**Durable lesson:** a covariance/correlation computation over a real feature matrix needs an explicit NaN policy stated out loud â€” the numpy default ("propagate to NaN, then someone downstream coerces NaN to 0") is silently equivalent to "assume uncorrelated," which is the least safe possible default for a dedup step. Separately: any "keep everything" fallback is only a safety net if it's loud: a silent one is indistinguishable from "the filter worked."

## Mixed categorical + numeric IV screening KeyError (0.3.14, fixed 0.3.15)

- **Trigger:** `selection_enabled=True` + shared WOE bins (`iv_use_woe_bins=True`) + `categorical_features` passed to `MonotoneWOEBinner(cate_feats=...)` + at least one categorical feature survives IV cut alongside numeric features in the same `high_iv_varlist`.
- **Symptom:** `KeyError: "['MIN', 'MEAN', 'MAX'] not in index"` in `WOE_Engine_Feature_Patch._screening_summary_from_engine` (or the parallel path in `Feature_Insights.get_var_analysis_report`) when merging `proc_means_by_grp` output. Pure-numeric feature sets unaffected.
- **Root cause:** `proc_means_by_grp` melts numeric + object columns together; pandas `describe()` on the mixed melt returns `count/unique/top/freq` for object columns, not `MIN/MEAN/MAX`.
- **Fix (0.3.15):** `Distribution_Tool.proc_means_for_screening` splits by dtype before calling `proc_means_by_grp`; categorical rows get `min/mean/max` as NaN.

**Durable lesson:** any batch stats helper that assumes numeric `describe()` columns will break the moment categorical features share a melt with numerics â€” split by dtype or reindex defensively.

## RejectInferencePipeline weighted GBM training was a no-op (fixed 0.3.16)

- **Trigger:** any RI method that emits `_weight` (`fuzzy_augment`, `parceling`) with `train_ri_models=True`. Approved-only methods (`simple_augment`, `hard_cutoff`) are unaffected.
- **Symptom:** none â€” training succeeds, no warning, no error. Model output looks plausible. But the weighted rows have zero effect: swapping in any other weight vector (all-ones, random, business rebalance) trains an identical model to the true `_weight`.
- **Root cause (`Pipeline/reject_inference.py` L623 in 0.3.15):** `fit_kwargs = {"wgt": weight}` passed to `GradientBoostingModel.fit`. The fit signature is `(x, y, valx, valy, init_score=None, sample_weight=None, eval_sample_weight=None, sample_weight_eval_set=None, **kwargs)` â€” `wgt` is not a real parameter and is silently absorbed by `**kwargs` and dropped. `GradientBoostingModel` uses `sample_weight`. Old SMF versions did expose `wgt=` on LGB fits, but the API was unified to `sample_weight` in an earlier version and this pipeline site was never updated.
- **Fix (0.3.16):** `fit_kwargs = {"sample_weight": weight}`. Regression test `SuperModelingFactory_pytest/test_reject_inference_weighted.py` intercepts `GradientBoostingModel.fit` and asserts (a) `sample_weight` is in the observed kwargs, (b) `wgt` never is, (c) the vector length matches the train row count.

**Durable lesson:** any `**kwargs`-accepting fit / call site where a Pipeline hand-rolls the kwarg dict is a silent-drop trap when the underlying signature evolves. When SMF unifies a param name (as with `wgt` â†’ `sample_weight`), grep every callsite in `Pipeline/*.py` at the same time, not just the `Model/*.py` definition â€” the compiler will not catch drift into `**kwargs`. `require_params(fit, "sample_weight")` in the pytest suite would catch this class of regression.

## Missing-rate screening stage gap (fixed 0.3.15)

- **Before 0.3.15:** `FeatureScreenConfig` and `FeatureValidationPipelineConfig` had no `missing_rate_threshold`; screening ran PSI â†’ IV â†’ corr only. `missing_rate` appeared in IV tables but was not a pre-PSI filter stage.
- **Fix (0.3.15):** `missing_rate_threshold: float | None = None` on both configs; when set, stage 0 runs on INS before PSI, drops features above threshold, persists `missing_rate_table` / `missing_rate_dropped` on `WeightedScreenResult` and FVP selection outputs.

## `apply_woe` O(nÂ²) performance (fixed before 0.3.3)

`WOE_Monotone_Binner.apply_woe` inserted new WOE columns one at a time via `df[col] = ...`, each insertion forcing pandas to reallocate its internal block manager. On wide batches (100â€“250+ new columns per call) this dominated runtime and was the confirmed (via `py-spy` stack sampling of a live, stuck process) root cause of a 3,380-candidate-feature validation run blowing through a 12-hour cell timeout with 14/14 batches still not finished. Fixed by collecting the new columns and adding them via a single `pd.concat(axis=1)`.

**Durable lesson:** if a WOE/feature-assembly step scales worse than linear with column count, suspect column-by-column DataFrame mutation before suspecting the statistics.

## Persistence

- **Always load SMF-saved models via `Core.load_model(path, return_metadata=True)`.** A bare `pickle.load` on an SMF model artifact raises a confusing `No module named '1'`-shaped error â€” SMF wraps saved models in its own versioned container, not a plain pickle of the estimator.
- **`LRMaster` used to pickle its full training frame (`_data`)** â€” tens to hundreds of MB, and literally row-level customer data riding along inside a model artifact (a real governance issue, not just bloat). Fixed via `__getstate__`/`__setstate__` dropping `_data` on save. Consequence: `get_statsmodel_summary` / `get_aic` / `get_bic` called on a *loaded* model now require an explicit `data=` kwarg and raise a clear `ValueError` telling you so if you omit it â€” this is intentional, not a regression.
- **`CreditModelPipelineConfig` had no `save_models` field at all** before it was added â€” the `models/` output directory was created either way (dead giveaway that something's missing, not that nothing was supposed to be there) but stayed empty until the field landed.

## Evaluation-table labeling

- **`model_perf_compare`'s `sample_name` was hardcoded to `"oot"`** regardless of what population was actually being scored â€” every table it powers (including every one of `ScoreComparisonPipeline`'s global and group-by outputs) had its `index`/scope column mislabeled `"oot"` even for the full sample or an arbitrary subgroup. Fixed by adding an explicit `sample_name` parameter; `ScoreComparisonPipeline` now passes `"global"` at every internal call site, and the bare function's own default changed to the neutral `"all"`. If anything calls `model_perf_compare` directly without passing `sample_name`, its output labeling changed meaning across this version bump â€” check call sites, not just the pipeline.

## Explainability

- **Owen values deliberately skip `xgb`** â€” `_run_explainability`'s Owen branch is gated `name != "xgb"`. By design, not a bug; don't "fix" it without finding out why first.
- **LIME was fully hand-rolled through 0.3.6**, absorbed natively into `Explainability.Model_Explainer` (`lime_explain_instance`, `lime_global_importance`, with built-in missing-value handling) as of **0.3.13**. This is the concrete precedent for the escalation ladder in `SKILL.md`: a hand-built workaround (including its NaN-median-fill fix for the fact that raw/non-WOE features commonly contain NaNs, which vanilla LIME's sampler can't handle â€” `scipy.stats.truncnorm: scale must be positive`) can be quietly absorbed upstream. **Check `ModelExplainer` for a native method before hand-rolling an explainer again on any version â‰¥0.3.13.**
- **`explain_outputs` (the computed SHAP/Owen tables) may not be persisted to disk** even when an `explain/` output directory is created and `explain_models`/`owen_enabled` ran without error â€” the tables can live only in the in-memory `CreditModelPipelineResult` for that kernel's lifetime. If the kernel dies before you persist them yourself (a later cell crashing, an OOM, a LIME NaN blowup), they're gone; recover by reloading the saved model via `Core.load_model(..., return_metadata=True)` (for `feature_cols`/`feature_source`) and recomputing with the same `sample_n`/`background_n`/`random_state`. Check whether the installed version's `_run_explainability` calls a persistence helper (look for something like `persist_explain_outputs` in `Pipeline._common` actually being invoked, not just importable) before assuming this gap is still open.
- **Environment, not SMF:** a long stretch of this project ran with `explain_models=[]` as a standing workaround because the installed `numba` (0.56.4, compiled against numpy<1.24) was ABI-incompatible with the installed `numpy` (2.2.6) â€” `import shap` crashed transitively via numba. `_run_explainability` catches this per-model and returns `{"error": repr(exc)}` rather than failing the whole pipeline, which can look like "explainability isn't configured" rather than "the environment is broken" â€” if a model's `explain_outputs` entry has an `"error"` key, check the environment before the config. Working combination found: `numba==0.66.0` + `llvmlite==0.48.0` + `lime==0.2.0.1` + `scikit-image>=0.25` + `pywavelets>=1.8`, with numpy/shap/lightgbm/xgboost/catboost left untouched.

## Data hygiene that belongs *before* the SMF call, not inside one

Neither of these is an SMF bug â€” both are exactly the kind of requirement that feels like "I'll just hand-filter the DataFrame" until you check whether a Config field already covers it:

- **Negative sentinels in CDC feature/score columns** (`-1`, `-2`, `-100`, observed at ~35% of an entire feature matrix's cells by count) are not valid numeric values. SMF's binners will bin them as real numbers if you feed them in as-is â€” always `NaN` them first, and do it consistently for every column an evaluation touches (score columns included, not just model features).
- **A population that should be excluded from fitting but retained for evaluation** (this project's "FT" cohort â€” a natural-approval segment that bypasses normal model scoring) is handled by `woe_fit_query` (fit-time exclusion) + `extra_eval_datasets` (eval-only inclusion) on the relevant pipeline, once those fields exist on the installed version â€” check before hand-splitting the DataFrame yourself.

## Pipeline hardening batch (fixed 0.3.17)

Four independent HIGH-severity gaps found during a full re-read of the seven Pipeline modules on the 0.3.15 â†’ 0.3.16 baseline, all shipped together in 0.3.17. None of them threw a loud exception before the fix; all four either produced silently-wrong outputs or reported an error a long way from the actual root cause.

- **H1 â€” `CreditModelPipeline._fit_models` mutated the user's `cfg.model_params["lr"]`** on the LR training branch. The old line was `lr = LRMaster(params=params or None, standardize=bool(params.pop("standardize", False)) if params else False)`; `params.pop("standardize", False)` ran on the *same* dict already passed to `LRMaster(...)` (via reference), and the truthiness short-circuit made the behavior further order-dependent. Symptom: user's config `{"standardize": True, ...}` had `"standardize"` disappear after `.run()`, breaking a second run or any downstream code that expected the field. Fix: copy first, pop second â€” same pattern the neighboring `_run_lr_search` already used at L901â€“902.
- **H2 â€” `oot_col == 0` silently routed every string-flag row to OOT.** Both `CreditModelPipeline._split_data` (L420â€“421 pre-fix) and `FeatureValidationPipeline._split_dataset` (L910â€“912 pre-fix) used `work[cfg.oot_col] == 0` / `!= 0`. If the column came in as `"0"`/`"1"` strings or `"train"`/`"test"` labels (common Hive/ODPS export artifact), the equality comparison against integer `0` was false for every row â†’ `ins_oos` came out empty, `oot` swallowed the whole frame, no exception. Fix: shared `split_oot_by_flag` helper in `Pipeline/_common.py` runs `pd.to_numeric(..., errors="raise")` â€” numeric strings (`"0"`/`"1"`) still work, boolean flags still coerce, but non-numeric content now raises a clear `TypeError` at the split step instead of producing empty training data.
- **H3 â€” `RejectInferencePipeline` implicit prescore path was un-validated.** `_run` triggers prescore training either when `cfg.train_prescore=True` **or** when `cfg.score_col not in data.columns` (implicit path â€” no external score means we have to fit one). `_validate_input` only checked `target_col` presence on the first branch, so the second branch failed deep inside `_fit_prescore` with a positionally confusing `KeyError`. Fix: `_validate_input` now checks `cfg.train_prescore or cfg.score_col not in data.columns` up-front and raises with a message that includes both `train_prescore` and the `score_col` presence status.
- **H4 â€” `RejectInferencePipeline` `oot_frac + ri_validation_frac >= 1` silently leaked OOT into validation.** `_sample_validation_ids` builds `pool = approved[~approved["_smf_ri_row_id"].isin(exclude_ids)]`; when `exclude_ids` (OOT + prior selections) covered everything, `pool` was empty and the code fell back to `pool = approved.copy()` â€” meaning the "held-out" validation set was resampled from the full approved population, including the OOT rows. Metrics on validation and OOT then shared rows, quietly biasing the validation number toward OOT and invalidating any early-stopping decision that used it. Fix: `_validate_ri_approved_config` now enforces `oot_frac + ri_validation_frac < 1` and `oot_frac âˆˆ [0, 1)` at `run()` start; the fallback branch itself was left alone because the assertion makes it unreachable under any valid config.

**Durable lessons across all four:**
- Silent equality-based routing on an ambiguously-typed column (`work[col] == 0`) is a landmine any time the column might be a string/boolean; always coerce and validate at the boundary.
- "Fallback to full population" is only safe if it's loud â€” the RI validation fallback and the string-oot-flag routing both silently produced business-plausible-looking but wrong numbers.
- When one code path validates a precondition and a parallel path skips validation, the "unchecked" path is where the confusing error will surface â€” always check every branch that requires the precondition, not just the obvious one.

## Pipeline hygiene batch (fixed 0.3.18)

Three independent MEDIUM-severity gaps found in the same 0.3.15 â†’ 0.3.17 review that produced the H1â€“H4 batch, held back for a follow-up release to keep 0.3.17 focused. None crashed loudly on the fix path; each either produced silently-wrong data, silently discarded user intent, or failed at a confusing point far from the real problem.

- **M2 â€” `CreditModelPipeline._fit_woe` copied `warm_start_score_col` with a bare `.to_numpy()` and no length guard.** Both the reused-from-screening branch (L600â€“606 pre-fix) and the fresh-fit branch (L697â€“704 pre-fix) did `df[cfg.warm_start_score_col] = splits[name][cfg.warm_start_score_col].to_numpy()` to copy the warm-start scores from raw splits into WOE-transformed splits. This relied on WOE transformation preserving the row count exactly; any upstream `.dropna()`, fit-query filter, or reindex on either side would silently corrupt the join â€” one frame's rows attached to another frame's scores, no error, no warning. Fix: new `Pipeline/_common.copy_column_length_checked(dst, src, col, dst_name, src_name)` helper enforces `len(dst) == len(src)` before the positional `.to_numpy()` copy and raises `ValueError` with both frame names if they've drifted. Both `_fit_woe` call sites and the `_reuse_screening_woe` helper now route every warm-start copy through it.
- **M3 â€” `RejectInferencePipeline._run` silently overwrote a user-supplied `cfg.score_col`.** The prescore path fires when `cfg.train_prescore=True` **or** when `cfg.score_col not in data.columns`. In the combined case (`train_prescore=True` *and* `score_col` already present in the input frame) `_fit_prescore` unconditionally wrote its own model's predictions into that column with no warning â€” the user's pre-supplied score column vanished with no trace. Fix: emit a `UserWarning` before calling `_fit_prescore` when the overwrite is about to happen, naming the column and both opt-outs (`train_prescore=False` to reuse, or rename `cfg.score_col` to sidestep the collision). Skipping the prescore fit was rejected because a user who explicitly set `train_prescore=True` *is* asking for a fresh fit â€” the bug is only that the collision was invisible, not that the overwrite was inherently wrong.
- **M4 â€” `_fit_prescore` fed an empty frame to `SampleSplitter.split_df`.** `approved = data[(approved_col==1) & target_col.notna()]` with no guard: if no approved rows had observed targets (mislabeled columns, an all-rejected batch, target-column contract mismatch), `SampleSplitter.split_df` failed deep inside sklearn with a confusing `stratify` error. Fix: `_fit_prescore` now raises a targeted `ValueError` when `len(approved) == 0` with counts of `approved` and `target-observed` rows plus the total row count â€” the message tells the caller whether they have an approved-count problem, a target-null problem, or both. Aligns with the H3 pattern of failing at input validation with diagnostics rather than deep in a downstream call.

**Durable lessons across all three:**
- A positional column copy (`.to_numpy()`, `.values`) is only safe if the length invariant is enforced explicitly â€” pandas won't error on a length mismatch because there is no length mismatch at that call, the mismatch is between the two frames being combined and pandas never sees them together.
- "Overwrite silently on collision" is a defensible default only when the collision itself is impossible; the moment two config paths (`train_prescore`, `score_col`) can both be set, the collision case needs a warning at minimum.
- Any deep-in-sklearn error a Pipeline can predict from its own preflight state should raise early with counts, not later with a stack trace pointing at library internals.

## Pipeline hygiene batch #2 (fixed 0.4.0)

Six independent MEDIUM-severity gaps across five Pipeline modules, shipped together. One of them (**M5**) is a behavior change that flips a default and forced the version bump from a 0.3.19 patch to a 0.4.0 minor. The other five are pure hardening â€” silent-failure closures, one performance rewrite, one preview API.

- **M1 â€” `CreditModelPipeline._run_optuna` fell back to defaults on empty `search_spaces={}`.** The old line was `search_spaces = cfg.optuna_params.get("search_spaces") or self._default_search_spaces()` â€” an explicit empty dict is falsy, so `or` swapped it out for the built-in default. Users passing `optuna_params={"search_spaces": {}}` to say "run no optuna search" got the default search anyway. Fix: `is None` check instead of truthiness. Untangling the same footgun in any newly written pipeline code is now cheap because the pattern is fresh.
- **M5 â€” `ScoreComparisonPipelineConfig.cross_vars` default flipped from `["rating"]` to `[]`.** The old default forced a `rating` column via `_validate_input` and crashed for the (majority) callers who don't have one. In 0.4.0 the default is `[]`, `_validate_input` no longer includes `cross_vars` in the required-columns list, and the run loop soft-skips any `cross_var` that isn't in the input frame with a `_logger.warning`. **Breaking change**: callers who relied on the implicit rating breakdown must now pass `cross_vars=["rating"]` explicitly. This is why the release is 0.4.0 rather than 0.3.19.
- **M6 â€” `ScoreConsistencyUATPipeline` silently coerced object columns to numeric.** `_load_dataframes` ran `pd.to_numeric(df[col], errors="coerce")` on every object column and swapped in the coerced version whenever `notna().any()`. Rating columns like `["A", "B", "1", "2"]` silently lost the alpha buckets to NaN. Fix: new `numeric_coercion_mode` config field with three modes (`"safe"` default = only coerce when â‰¥`numeric_coercion_min_ratio` of non-null values parse, warn otherwise; `"aggressive"` = old 0.3.x behavior but with a warning per lossy column; `"off"` = never coerce). To restore 0.3.x behavior explicitly: `numeric_coercion_mode="aggressive"`.
- **M7 â€” `FeatureValidationPipeline._split_frame` bare-except swallowed stratify failures.** `try: SampleSplitter(...).split_df(...) except Exception: pass` dropped every failure into a naive random split with no diagnostic. Fix: `except Exception as exc` + `_logger.warning` that includes `target`, `test_size`, `random_state`, and the repr of the exception. No behavior change to the fallback itself; only observability. Module now imports `logging` and defines `_logger = logging.getLogger(__name__)`.
- **M8 â€” `FeatureValidationPipeline._high_corr_pairs` was O(nÂ²) Python with `.loc[]`.** The old nested loop did ~4.5M scalar `corr_matrix.loc[var1, var2]` lookups at 3000 features. Rewritten with `np.triu_indices(n, k=1)` to lift the upper triangle in one vectorized shot, `np.isfinite & (np.abs > threshold)` mask, then set-membership for `pair_type`. **Output is numerically identical** to the loop â€” verified by the regression test `test_pipeline_medium_0400.py::TestM8HighCorrPairsVectorized::test_matches_loop_reference` which keeps a copy of the loop as a numerical oracle. No behavior change, just faster.
- **M9 â€” `SampleAnalysisPipeline` silently spawned up to 960 splits by default.** Default `4 targets Ã— 4 windows Ã— 3 ratios Ã— 20 seeds` grid Ã— `SampleSplitter` per cell, no preview. Fix: new `dry_run: bool = False` config field short-circuits `run()` to skip all computation and only return an estimate dict in `split_candidate_summary`; new public method `estimate_split_count(data=None) -> dict` returns the grid product plus, when `data` is provided, `available_oot_periods` and `windows_usable`. Nothing in the default execution path changed â€” the preview is strictly opt-in.

**Durable lessons across the batch:**
- **`or` on a container defaults to non-empty**: `cfg.get("x") or default` treats a deliberate `{}` / `[]` / `""` the same as "unset." When a user's intent might be "explicitly empty," write `is None` instead. Applies to M1, but grep every Pipeline config-consumer for the same idiom before it lands again.
- **A default value that requires a column is a footgun, not a feature.** `cross_vars=["rating"]` (M5) is a two-liner in the config file that quietly costs a KeyError to every user without the specific rating column the author had in mind. Any list-typed config default should be `[]` unless the column is one SMF itself creates.
- **`pd.to_numeric(..., errors="coerce")` followed by a truthiness check is a silent lossy conversion.** M6's fix (opt-in mode + per-column loss warn) is the pattern to reach for any time an object column might contain a mix of parseable numbers and legitimate string labels.
- **`np.triu_indices` is the vectorization for anything shaped like a pair-loop over a symmetric matrix** (M8). If a Pipeline method has `for i, ...: for j in ... [i+1:]: matrix.loc[i, j]`, it's a candidate.
- **A grid-search default that multiplies to hundreds of runs needs a `dry_run` or `estimate_*` counterpart on the same config** (M9). Adding it as a public method (not just a private helper) keeps the escalation ladder from having to re-derive the same math on the caller side.
